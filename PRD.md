# Product Requirements Document (PRD)

## 1. ภาพรวมผลิตภัณฑ์

### 1.1 ชื่อระบบ

ระบบจัดการการส่งงานรายวิชาของนักศึกษา

### 1.2 คำอธิบายโดยย่อ

เว็บแอปพลิเคชันสำหรับจัดการข้อมูลการมอบหมายงานและการส่งงานของนักศึกษาในรายวิชาต่างๆ โดยรองรับการส่งงานได้หลายรูปแบบ เช่น ไฟล์ PDF, รูปภาพ, ลิงก์ และวิดีโอ พร้อมระบบจัดการสิทธิ์ผู้ใช้งาน 3 บทบาท ได้แก่ Admin, Teacher และ Student

### 1.3 เป้าหมายของระบบ

- ช่วยให้อาจารย์สร้างและจัดการงานในแต่ละรายวิชาได้สะดวก
- ช่วยให้นักศึกษาส่งงานและติดตามสถานะการส่งงานได้ชัดเจน
- ช่วยให้ผู้ดูแลระบบจัดการผู้ใช้งาน รายวิชา และข้อมูลพื้นฐานของระบบได้
- ลดปัญหาการส่งงานกระจัดกระจายผ่านหลายช่องทาง
- ทำให้การตรวจสอบงานและติดตามกำหนดส่งเป็นระบบมากขึ้น

### 1.4 กลุ่มผู้ใช้งานหลัก

- Admin: ผู้ดูแลระบบ
- Teacher: อาจารย์ผู้สอน
- Student: นักศึกษา

## 2. ขอบเขตของระบบ

### 2.1 ฟีเจอร์ที่อยู่ในขอบเขต

- ระบบเข้าสู่ระบบและจัดการบัญชีผู้ใช้
- ระบบบทบาทและสิทธิ์การใช้งาน
- ระบบจัดการรายวิชา
- ระบบจัดการการลงทะเบียนเรียนหรือการเข้าร่วมรายวิชา
- ระบบสร้างและจัดการงานมอบหมาย
- ระบบส่งงานของนักศึกษา
- ระบบอัปโหลดไฟล์ PDF และรูปภาพ
- ระบบแนบลิงก์และวิดีโอ
- ระบบตรวจสอบสถานะการส่งงาน
- ระบบให้คะแนนหรือข้อเสนอแนะเบื้องต้น
- ระบบแจ้งเตือนงานใกล้ครบกำหนดหรือเลยกำหนด
- หน้า Dashboard ตามบทบาทผู้ใช้งาน

### 2.2 ฟีเจอร์ที่ยังไม่อยู่ในขอบเขตระยะแรก

- ระบบประชุมออนไลน์
- ระบบแชตแบบ real-time
- ระบบ AI ตรวจ plagiarism
- ระบบชำระเงิน
- ระบบ live streaming
- ระบบ mobile app แบบ native

## 3. บทบาทผู้ใช้งานและสิทธิ์

### 3.1 Admin

Admin คือผู้ดูแลระบบทั้งหมด มีหน้าที่จัดการข้อมูลหลักและควบคุมสิทธิ์การใช้งาน

สิทธิ์หลัก:

- เข้าสู่ระบบ Admin Dashboard
- สร้าง แก้ไข ปิดใช้งาน หรือลบผู้ใช้
- กำหนดบทบาทผู้ใช้เป็น Admin, Teacher หรือ Student
- จัดการข้อมูลรายวิชา
- กำหนด Teacher ให้กับรายวิชา
- ดูรายการงานและการส่งงานในระบบ
- จัดการข้อมูลพื้นฐาน เช่น ภาคเรียน ปีการศึกษา หมวดหมู่รายวิชา
- ดูสถิติภาพรวมของระบบ

### 3.2 Teacher

Teacher คืออาจารย์ผู้สอน มีหน้าที่ดูแลรายวิชาของตนเอง สร้างงาน และตรวจงานนักศึกษา

สิทธิ์หลัก:

- เข้าสู่ระบบ Teacher Dashboard
- ดูรายวิชาที่ตนเองรับผิดชอบ
- สร้าง แก้ไข และลบงานมอบหมายในรายวิชาของตนเอง
- กำหนดรายละเอียดงาน วันเริ่มส่ง และวันครบกำหนด
- กำหนดประเภทไฟล์หรือรูปแบบการส่งงานที่อนุญาต
- ดูรายชื่อนักศึกษาในรายวิชา
- ดูงานที่นักศึกษาส่ง
- ดาวน์โหลดไฟล์งานหรือเปิดลิงก์ที่นักศึกษาส่ง
- ให้คะแนนและข้อเสนอแนะ
- ดูสถานะการส่งงาน เช่น ส่งแล้ว ยังไม่ส่ง ส่งช้า

### 3.3 Student

Student คือนักศึกษาที่ลงทะเบียนในรายวิชา มีหน้าที่ดูงานที่ได้รับมอบหมายและส่งงานตามกำหนด

สิทธิ์หลัก:

- เข้าสู่ระบบ Student Dashboard
- ดูรายวิชาที่ตนเองเรียน
- ดูรายการงานมอบหมายในแต่ละรายวิชา
- ดูรายละเอียดงานและกำหนดส่ง
- ส่งงานในรูปแบบ PDF, รูปภาพ, ลิงก์ หรือวิดีโอ
- แก้ไขการส่งงานได้ก่อนหมดเขต หากอาจารย์อนุญาต
- ดูสถานะการส่งงานของตนเอง
- ดูคะแนนและข้อเสนอแนะจากอาจารย์

## 4. User Persona

### 4.1 Admin

- ต้องการระบบที่จัดการผู้ใช้และรายวิชาได้ง่าย
- ต้องการเห็นภาพรวมว่าระบบมีการใช้งานมากน้อยเพียงใด
- ให้ความสำคัญกับความถูกต้องของข้อมูลและการควบคุมสิทธิ์

### 4.2 Teacher

- ต้องการสร้างงานได้รวดเร็วและระบุเงื่อนไขการส่งงานได้ชัดเจน
- ต้องการตรวจงานได้สะดวก ไม่ต้องรวบรวมไฟล์จากหลายช่องทาง
- ต้องการเห็นสถานะว่านักศึกษาคนใดส่งแล้วหรือยังไม่ส่ง

### 4.3 Student

- ต้องการรู้ว่างานใดต้องส่ง เมื่อไร และส่งอย่างไร
- ต้องการส่งงานได้ง่ายจากอุปกรณ์ที่ใช้อยู่
- ต้องการตรวจสอบได้ว่างานถูกส่งสำเร็จแล้ว

## 5. User Journey หลัก

### 5.1 Teacher สร้างงานมอบหมาย

1. Teacher เข้าสู่ระบบ
2. เลือกรายวิชาที่ต้องการ
3. เปิดหน้ารายการงานมอบหมาย
4. กดสร้างงานใหม่
5. กรอกชื่องาน รายละเอียด คะแนนเต็ม วันที่เปิดรับ และวันครบกำหนด
6. เลือกรูปแบบการส่งงานที่อนุญาต เช่น PDF, รูปภาพ, ลิงก์, วิดีโอ
7. บันทึกงานมอบหมาย
8. ระบบแสดงงานใหม่ให้นักศึกษาในรายวิชาเห็น

### 5.2 Student ส่งงาน

1. Student เข้าสู่ระบบ
2. เปิด Dashboard เพื่อดูงานที่ต้องส่ง
3. เลือกรายวิชาและงานที่ต้องการส่ง
4. อ่านรายละเอียดงานและตรวจสอบกำหนดส่ง
5. เลือกรูปแบบการส่งงาน
6. อัปโหลดไฟล์หรือกรอกลิงก์
7. กดยืนยันการส่งงาน
8. ระบบบันทึกเวลาและแสดงสถานะว่าส่งสำเร็จ

### 5.3 Teacher ตรวจงาน

1. Teacher เข้าสู่ระบบ
2. เลือกรายวิชา
3. เลือกงานมอบหมายที่ต้องการตรวจ
4. ดูรายการนักศึกษาและสถานะการส่งงาน
5. เปิดไฟล์หรือลิงก์งานที่นักศึกษาส่ง
6. ให้คะแนนและข้อเสนอแนะ
7. บันทึกผลการตรวจ
8. Student สามารถดูผลการตรวจได้

## 6. Functional Requirements

### 6.1 Authentication และ Authorization

- ผู้ใช้ต้องเข้าสู่ระบบก่อนใช้งาน
- ระบบต้องแยกสิทธิ์ตามบทบาท Admin, Teacher และ Student
- ผู้ใช้แต่ละบทบาทต้องเห็นเฉพาะเมนูและข้อมูลที่ตนเองมีสิทธิ์เข้าถึง
- ระบบต้องป้องกันการเข้าถึงข้อมูลข้ามบทบาทผ่าน URL โดยตรง
- ระบบควรมีฟังก์ชันออกจากระบบ

### 6.2 User Management

- Admin สามารถสร้างบัญชีผู้ใช้ใหม่ได้
- Admin สามารถแก้ไขข้อมูลผู้ใช้ เช่น ชื่อ อีเมล รหัสนักศึกษา หรือรหัสอาจารย์
- Admin สามารถกำหนดบทบาทให้ผู้ใช้ได้
- Admin สามารถปิดใช้งานบัญชีผู้ใช้ได้
- ระบบต้องไม่อนุญาตให้อีเมลหรือรหัสประจำตัวซ้ำกัน

### 6.3 Course Management

- Admin สามารถสร้างรายวิชาได้
- รายวิชาต้องมีข้อมูลอย่างน้อย:
  - รหัสวิชา
  - ชื่อวิชา
  - คำอธิบายรายวิชา
  - ภาคเรียน
  - ปีการศึกษา
  - Teacher ผู้รับผิดชอบ
- Teacher สามารถดูเฉพาะรายวิชาที่ตนเองรับผิดชอบ
- Student สามารถดูเฉพาะรายวิชาที่ตนเองลงทะเบียนหรือถูกเพิ่มเข้าไป

### 6.4 Enrollment Management

- Admin หรือ Teacher สามารถเพิ่ม Student เข้าในรายวิชาได้
- ระบบควรสามารถค้นหานักศึกษาด้วยชื่อ อีเมล หรือรหัสนักศึกษา
- Student ที่ไม่ได้อยู่ในรายวิชาไม่ควรเห็นงานของรายวิชานั้น
- ระบบควรมีสถานะการลงทะเบียน เช่น active, removed หรือ completed

### 6.5 Assignment Management

- Teacher สามารถสร้างงานมอบหมายในรายวิชาของตนเองได้
- Assignment ต้องมีข้อมูลอย่างน้อย:
  - ชื่องาน
  - รายละเอียดงาน
  - รายวิชา
  - คะแนนเต็ม
  - วันที่เปิดรับงาน
  - วันครบกำหนด
  - รูปแบบการส่งงานที่อนุญาต
  - สถานะเผยแพร่
- Teacher สามารถแก้ไขงานก่อนหรือหลังเผยแพร่ได้
- Teacher สามารถปิดรับงานด้วยตนเองได้
- ระบบต้องแสดงสถานะงาน เช่น ยังไม่เปิดรับ, เปิดรับ, ปิดรับ, เลยกำหนด

### 6.6 Submission Management

- Student สามารถส่งงานได้ตามรูปแบบที่ Teacher กำหนด
- รูปแบบการส่งงานที่รองรับ:
  - PDF
  - รูปภาพ เช่น JPG, JPEG, PNG, WEBP
  - ลิงก์ URL
  - วิดีโอ โดยอาจเป็นไฟล์วิดีโอหรือ URL วิดีโอ ขึ้นอยู่กับข้อจำกัดระบบ
- ระบบต้องบันทึกเวลาที่ส่งงาน
- ระบบต้องแสดงสถานะการส่งงาน:
  - ยังไม่ส่ง
  - ส่งแล้ว
  - ส่งช้า
  - ตรวจแล้ว
- หากส่งหลังวันครบกำหนด ระบบต้องระบุว่าเป็นการส่งช้า
- ระบบควรรองรับการส่งงานหลายไฟล์ในหนึ่ง assignment หาก Teacher อนุญาต
- ระบบควรมีการตรวจสอบชนิดไฟล์และขนาดไฟล์ก่อนอัปโหลด

### 6.7 Grading และ Feedback

- Teacher สามารถให้คะแนนงานที่นักศึกษาส่งได้
- Teacher สามารถใส่ข้อเสนอแนะเป็นข้อความได้
- คะแนนต้องไม่เกินคะแนนเต็มของ assignment
- Student สามารถดูคะแนนและ feedback ของตนเองได้
- Student ไม่สามารถดูคะแนนของผู้อื่นได้

### 6.8 Notification

- ระบบควรแจ้งเตือน Student เมื่องานใหม่ถูกเผยแพร่
- ระบบควรแจ้งเตือน Student เมื่องานใกล้ครบกำหนด
- ระบบควรแจ้งเตือน Teacher เมื่อมี Student ส่งงาน
- ในระยะแรก การแจ้งเตือนอาจแสดงในระบบก่อน ยังไม่จำเป็นต้องส่งอีเมล

### 6.9 Dashboard

#### Admin Dashboard

- จำนวนผู้ใช้ทั้งหมด
- จำนวน Teacher
- จำนวน Student
- จำนวนรายวิชา
- จำนวน assignment ทั้งหมด
- จำนวน submission ทั้งหมด

#### Teacher Dashboard

- รายวิชาที่รับผิดชอบ
- งานที่กำลังเปิดรับ
- งานที่รอตรวจ
- สถิติการส่งงานของแต่ละ assignment

#### Student Dashboard

- งานที่ต้องส่ง
- งานใกล้ครบกำหนด
- งานที่ส่งแล้ว
- งานที่ตรวจแล้วพร้อมคะแนน

## 7. Non-Functional Requirements

### 7.1 Usability

- หน้าจอต้องใช้งานง่ายและเหมาะกับผู้ใช้ที่ไม่เชี่ยวชาญเทคโนโลยี
- ขั้นตอนการส่งงานควรสั้นและชัดเจน
- ระบบต้องแสดงข้อความยืนยันเมื่อส่งงานสำเร็จ
- ระบบต้องแสดง error message ที่เข้าใจง่ายเมื่ออัปโหลดไม่สำเร็จ

### 7.2 Performance

- หน้า Dashboard ควรโหลดได้ภายใน 3 วินาทีภายใต้ข้อมูลทั่วไป
- การอัปโหลดไฟล์ควรมี loading state หรือ progress indicator
- ระบบควรรองรับผู้ใช้พร้อมกันในระดับห้องเรียนหรือหลายรายวิชา

### 7.3 Security

- รหัสผ่านต้องถูกจัดเก็บแบบ hash
- ต้องตรวจสอบสิทธิ์ทุกครั้งก่อนเข้าถึงข้อมูล
- ไฟล์ที่อัปโหลดต้องถูกจำกัดชนิดไฟล์
- ระบบต้องป้องกันการอัปโหลดไฟล์อันตราย
- URL ที่นักศึกษาส่งควรถูก validate
- ข้อมูลคะแนนและ submission ต้องเข้าถึงได้เฉพาะเจ้าของข้อมูลและผู้มีสิทธิ์

### 7.4 Reliability

- ระบบต้องไม่ทำให้ข้อมูล submission หายหลังจากส่งสำเร็จ
- ควรมีการสำรองข้อมูลตามรอบเวลาที่กำหนด
- หากอัปโหลดไฟล์ล้มเหลว ระบบต้องไม่สร้าง submission ที่ข้อมูลไม่ครบถ้วน

### 7.5 Compatibility

- รองรับการใช้งานบน desktop, tablet และ mobile browser
- รองรับ browser สมัยใหม่ เช่น Chrome, Edge, Firefox และ Safari
- UI ต้องเป็น responsive design

## 8. UI/UX Requirements

### 8.1 โทนสี

ระบบควรใช้โทนสีหลักตาม requirement:

- สีขาว: ใช้เป็นพื้นหลังหลัก เพื่อความสะอาด อ่านง่าย
- สีน้ำเงิน: ใช้เป็นสีหลักของปุ่ม เมนู และ element สำคัญ
- สีฟ้า: ใช้เป็นสีรองสำหรับ highlight, badge หรือสถานะทั่วไป
- สีม่วง: ใช้เป็น accent color สำหรับจุดเน้น เช่น กราฟ แท็ก หรือสถานะพิเศษ

### 8.2 แนวทางการออกแบบ

- ดีไซน์ควรดูสะอาด ทันสมัย และเป็นระบบ

- Dashboard ควรเน้นการสแกนข้อมูลอย่างรวดเร็ว
- ปุ่ม action หลักควรเด่นและอยู่ในตำแหน่งที่คาดเดาได้
- สถานะงานควรใช้สีและ label ที่ชัดเจน เช่น ส่งแล้ว, ยังไม่ส่ง, ส่งช้า, ตรวจแล้ว
- ฟอร์มควรมี validation message ใกล้ช่องกรอกข้อมูล

## 10. Business Rules

- Student ต้องอยู่ในรายวิชาก่อนจึงจะเห็น assignment ของรายวิชานั้น
- Teacher ต้องเป็นเจ้าของรายวิชาหรือได้รับสิทธิ์ก่อนจึงจะแก้ไข assignment ได้
- Student หนึ่งคนควรมี submission ล่าสุดเพียงชุดเดียวต่อ assignment เว้นแต่ระบบออกแบบให้เก็บ version history
- หากส่งงานหลัง due_at ให้สถานะเป็น late submission
- Teacher สามารถตรวจได้เฉพาะ submission ในรายวิชาที่ตนเองรับผิดชอบ
- คะแนนที่บันทึกต้องอยู่ระหว่าง 0 ถึง max_score
- Admin สามารถดูข้อมูลภาพรวมได้ แต่การแก้ไขคะแนนควรจำกัดเฉพาะ Teacher เว้นแต่มี policy เพิ่มเติม

## 11. Acceptance Criteria

### 11.1 Login และ Role

- เมื่อ Admin เข้าสู่ระบบ ต้องเข้าสู่ Admin Dashboard
- เมื่อ Teacher เข้าสู่ระบบ ต้องเข้าสู่ Teacher Dashboard
- เมื่อ Student เข้าสู่ระบบ ต้องเข้าสู่ Student Dashboard
- ผู้ใช้ไม่สามารถเข้าหน้าที่ไม่มีสิทธิ์ผ่าน URL โดยตรงได้

### 11.2 Assignment

- Teacher สามารถสร้าง assignment พร้อมกำหนดวันครบกำหนดได้
- Assignment ที่เผยแพร่แล้วต้องแสดงให้นักศึกษาในรายวิชาเห็น
- Assignment ที่ยังไม่เผยแพร่ต้องไม่แสดงให้นักศึกษาเห็น

### 11.3 Submission

- Student สามารถส่งงานด้วยไฟล์ PDF ได้
- Student สามารถส่งงานด้วยรูปภาพได้
- Student สามารถส่งงานด้วยลิงก์ได้
- Student สามารถส่งงานด้วยวิดีโอหรือลิงก์วิดีโอได้
- หลังส่งงานสำเร็จ ระบบต้องแสดงสถานะส่งแล้ว
- หากส่งหลังวันครบกำหนด ระบบต้องแสดงสถานะส่งช้า

### 11.4 Grading

- Teacher สามารถเปิดดู submission ของ Student ได้
- Teacher สามารถให้คะแนนและ feedback ได้
- Student สามารถดูคะแนนและ feedback ของตนเองได้
- Student ไม่สามารถดูคะแนนและ feedback ของผู้อื่นได้

## 12. MVP Scope

### 12.1 ฟีเจอร์ MVP ที่ควรทำก่อน

- Login และ role-based access
- Admin จัดการผู้ใช้
- Admin จัดการรายวิชา
- Teacher สร้าง assignment
- Student ดู assignment
- Student ส่งงานด้วย PDF, รูปภาพ และลิงก์
- Teacher ดู submission
- Teacher ให้คะแนนและ feedback
- Dashboard พื้นฐานของแต่ละ role

### 12.2 ฟีเจอร์หลัง MVP

- การแจ้งเตือนในระบบแบบละเอียด
- ส่งอีเมลแจ้งเตือน
- รองรับวิดีโอไฟล์ขนาดใหญ่
- ระบบ version history ของ submission
- ระบบ export คะแนน
- ระบบ comment thread ระหว่าง Teacher และ Student
- ระบบ bulk import นักศึกษาด้วยไฟล์ CSV หรือ Excel

## 13. ความเสี่ยงและประเด็นที่ต้องตัดสินใจ

- ต้องกำหนดขนาดไฟล์สูงสุดที่อนุญาตให้อัปโหลด
- ต้องตัดสินใจว่าจะเก็บวิดีโอเป็นไฟล์ในระบบ หรือให้ส่งเป็นลิงก์วิดีโอเท่านั้น
- ต้องกำหนดว่านักศึกษาสามารถแก้ไขงานหลังส่งแล้วได้หรือไม่
- ต้องกำหนดนโยบายการส่งงานหลัง deadline
- ต้องกำหนดว่า Admin มีสิทธิ์แก้คะแนนหรือไม่
- ต้องกำหนดวิธีนำ Student เข้ารายวิชา เช่น เพิ่มเอง, import, หรือใช้ course code

## 14. Assumptions

- ระบบนี้เป็นเว็บแอปพลิเคชันที่ใช้งานผ่าน browser
- ผู้ใช้ทุกคนต้องมีบัญชีในระบบก่อนใช้งาน
- รายวิชาถูกสร้างโดย Admin หรือผู้มีสิทธิ์
- Teacher เป็นผู้สร้างงานและตรวจงานในรายวิชาของตนเอง
- Student ส่งงานตาม assignment ที่ได้รับมอบหมาย
- การแจ้งเตือนระยะแรกเป็น notification ภายในระบบ

## 15. Success Metrics

- นักศึกษาสามารถส่งงานสำเร็จโดยไม่ต้องพึ่งช่องทางภายนอก
- Teacher สามารถดูสถานะการส่งงานของนักศึกษาได้ครบถ้วน
- จำนวนงานที่ส่งผิดช่องทางลดลง
- เวลาที่ใช้ในการรวบรวมงานของ Teacher ลดลง
- ผู้ใช้สามารถเข้าใจ workflow หลักได้โดยไม่ต้องมีคู่มือยาว
